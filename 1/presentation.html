用了什麼技術 / 工具

這個專案的「材料清單」

🎯

YOLOv8

AI 偵測模型本人。YOLO = You Only Look Once，「一眼看完整張圖」。預先用大量火災照片訓練過，已經「知道」火和煙長什麼樣。

⚙️

ONNX Runtime

AI 模型的執行引擎。把訓練好的模型轉成「通用格式 ONNX」，就能在任何電腦（Windows、Pi、手機）上跑，不需要 GPU。

🖼️

OpenCV

影像處理瑞士刀。負責讀影片、縮放、色彩轉換、畫框畫線、存檔——所有「跟畫面有關的事」都靠它。

🐍

Python

程式語言。把上面三個工具串在一起的「膠水」。語法簡單，AI 圈的主流語言。

🥧

Raspberry Pi 4B

運算硬體。一台跟信用卡差不多大的小電腦（約 1500 元台幣）。整套系統最終跑在它上面，不需要網路。

📦

INT8 量化

模型壓縮技術。把 AI 模型「縮小 4 倍」，讓便宜的小電腦也跑得動。壓縮過程有踩坑，後面會講。

兩種視覺化方式

同一個偵測結果，兩種畫法

📏 Bounding Box 預設

gif/result_detect.gif

煙霧偵測 ・ 矩形框畫法 ・ 紅框 = fire ／ 灰框 = smoke

最簡單的畫法：用「最小可包圍的矩形」圈住 AI 認定是火/煙的區域。 快速、直覺，但方框不能表達火焰的真實形狀。

🔥 Contour --contour

煙霧顏色輪廓（test1 前段）

在 YOLO 框框內，用 HSV 色彩空間 分析找出火焰橘紅色 / 煙霧灰白色的像素， 再用 OpenCV 描出真實輪廓——貼合形狀，僅多 4% 運算成本。

HSV 顏色範圍（visualize.py 定義）

🔥 fire：色相 0–35（黃→橘）＋ 160–180（紅），飽和度 ≥ 100，亮度 ≥ 100
💨 smoke：色相不限，飽和度 < 60（灰白色），亮度 ≥ 80
⚠️ 若 HSV 抓不到（夜景 / 黑煙）→ 自動退回方框，不會空白

06 / 08

開發歷程

從零到部署，走過的七個關卡

① 找模型 ・ 不從零訓練

使用 GitHub 上已用大量火災照片訓練好的 YOLOv8n 權重（.pt 檔）。省掉數週訓練時間。

② 匯出 ONNX 通用格式

PyTorch 格式 (.pt) 只能在有 PyTorch 的電腦上跑。轉成 ONNX 後，Pi 只需安裝 onnxruntime，不需要 PyTorch（省記憶體、省空間）。

③ 開發機驗證

先在 Windows + RTX 顯卡跑通，確認偵測正確、速度超過 100 FPS。

④ 部署 Raspberry Pi 4B

安裝 onnxruntime + opencv-headless（無桌面版本），FP32 模型在 Pi 上約 11 FPS，勉強達標。

⑤ INT8 量化（踩坑）

想加速 → 用 quantize_dynamic 把模型壓縮 4 倍。結果 Pi 直接報錯：「ARM CPU 不支援 ConvInteger 算子」。模型一個都跑不了。

⑥ 改用 QDQ 靜態量化（解決）

改用 quantize_static + QDQ + per_channel=False，產生 ARM 相容的 QLinearConv 算子。Pi 上順利跑起來，FPS 提升至 ~14。

⑦ 視覺化升級

加入 --contour，HSV 色彩分析讓標示貼合火焰真實形狀，只多 4% 運算成本。

效能數據

不同設定下能跑多快？

119開發機 ・ FP32 ・ 320
70開發機 ・ FP32 ・ 416
~11Pi 4B ・ FP32 ・ 320
~14Pi 4B ・ INT8 ・ 320

單位：FPS（每秒幾幀）。人眼覺得「流暢」的下限約 10 FPS。

📊 從這些數字學到什麼？

・ 桌機跑 320 → 119 FPS，遠超即時需求
・ Pi 跑同設定 → 11 FPS，算力約桌機的 1/10
・ INT8 量化後 → ~14 FPS（+30%），對 Pi 是明顯提升
・ 416 解析度在 Pi 上僅 7 FPS，低於目標，不建議

🎯 為什麼目標定在「10 FPS 以上」？

低於 10 FPS 會看起來卡頓、像慢動作。火災偵測不需要電玩等級的 60 FPS， 但至少 10 FPS 才能即時追蹤火勢蔓延方向。 FP32 模型在 Pi 上剛好達標（~11），INT8 則多了安全餘量（~14）。

① Letterbox 等比縮放

AI 模型只吃 320 × 320 的正方形圖，但影片通常是 640×480 的長方形——如果直接硬縮，火焰會被壓扁變形，AI 就認不出來了。

解法：等比縮放 + 灰邊填滿（像電影院的 letterbox 黑邊）

320 × 320

之後推論得到的框框座標，也要把灰邊「扣掉」、再換算回原始尺寸，才能在影片上畫在正確位置。

# detector.py 內的 _letterbox()  核心邏輯
scale = 320 / max(h, w)          # 縮放比例
nh, nw = h * scale, w * scale    # 縮後高寬
dw = (320 - nw) / 2              # 左右各填這麼多灰
dh = (320 - nh) / 2              # 上下各填這麼多灰

② Preprocess 前處理

把畫面轉成 AI 模型「看得懂」的數字格式。共三個動作：

  • BGR → RGB：OpenCV 讀進來的畫面色彩順序是藍-綠-紅（BGR），但 AI 訓練時用紅-綠-藍（RGB）。不換的話顏色辨識會全錯。
  • ÷ 255：像素值原本是 0–255 的整數，AI 喜歡 0–1 的小數。把所有數值除以 255，讓模型計算更穩定。
  • HWC → NCHW：OpenCV 的格式是「高×寬×通道（HWC）」，AI 模型要的是「批次×通道×高×寬（NCHW）」——用 NumPy 轉置，再加一個 batch 維度。

③ ONNX 模型推論

把前處理好的 Tensor 丟給 ONNX Runtime，等它算完後取回結果。這步是整個流程最耗時的部分。

  • 輸入形狀：[1, 3, 320, 320]（1張圖 × 3色彩通道 × 320×320 像素）
  • 輸出形狀：[1, 6, 2100]（2100 個候選框 × 每框 6 個數字）

2100 個候選框是怎麼來的？YOLOv8 把 320×320 的圖分成不同大小的格子（20×20、40×40、80×80），每個格子預測幾個框，加起來剛好是 2100。

每個候選框的 6 個數字：前 4 個是位置（中心 x, 中心 y, 寬, 高），後 2 個是「這裡是 fire 的機率」和「這裡是 smoke 的機率」。

④ 信心度過濾（Confidence Filter）

2100 個候選框裡，絕大多數 AI 自己也沒把握——例如天空、地面、牆壁上也會有低分候選框。

過濾規則：取每個框的 fire 分數和 smoke 分數中的較大值，若低於 --conf（預設 0.35）就丟掉。

→

  • 調低 --conf（如 0.25）→ 更多框通過，偵測更積極，但可能誤報
  • 調高 --conf（如 0.45）→ 更少框，偵測更保守，漏報較少
  • 對 FPS 的影響幾乎可忽略（<2%）
# detector.py  _postprocess()
confidences = scores.max(axis=1)          # fire 和 smoke 取較大值
mask = confidences >= self.conf_thres      # 低於門檻的直接去掉

⑤ NMS 去重疊（Non-Maximum Suppression）

信心度過濾後，還剩十幾個框，但 AI 常對同一個火焰畫好幾個重疊的框。NMS 的任務是只留最好的那一個。

NMS 步驟：

  • 1. 按信心度由高到低排序
  • 2. 從信心最高的框開始，計算它和其他框的重疊比例（IoU）
  • 3. 重疊超過 --iou（預設 0.45）的框 → 刪掉（認為是同一個物體的重複偵測）
  • 4. 對剩下的框重複 2-3 步驟

IoU（Intersection over Union）：兩個框交集面積 ÷ 聯集面積。IoU = 1 代表完全重疊，= 0 代表完全不重疊。

→

NMS 是逐類別執行的——fire 的框和 smoke 的框不會互相干擾。

⑥ 座標換算（回到原畫面）

AI 推論在 320×320 的空間做，但框要畫在原始影片（如 640×480）上。需要三步驟還原：

  • 1. 減去 letterbox padding：上下或左右的灰邊偏移量要扣掉
  • 2. 除以縮放比例：把 320 空間的座標放大回原始尺寸
  • 3. clip 限制範圍：防止框超出畫面邊界（負值或超過寬高）

① 開啟影片 ＋ 跳秒

用 OpenCV 的 VideoCapture 開啟影片檔（或攝影機串流）。

如果有 --start-sec，在開始讀幀之前先用 CAP_PROP_POS_MSEC 跳到指定時間點——適合測試 899 秒長影片的中段，不用等到那裡。

同時讀取影片的 FPS 資訊（用來設定輸出影片的幀率）。若 FPS 數值不合理（如損壞檔案給出 0 或 9999），自動 fallback 到 30。

cap = cv2.VideoCapture("test_videos/fire.mp4")
# 跳到第 450 秒
cap.set(cv2.CAP_PROP_POS_MSEC, 450 * 1000.0)
src_fps = cap.get(cv2.CAP_PROP_FPS)
# 異常值保護
src_fps = src_fps if 1.0 <= src_fps <= 120.0 else 30.0

② Speed 加速讀取

--speed N 讓程式每 N 幀只「真正讀」1 幀，輸出影片時長縮為 1/N。主要用於測試階段（快速確認長影片的偵測結果）。

技術重點：中間跳過的 N-1 幀用 cap.grab() 而非 cap.read()。

  • grab()：只抓取幀的指標，不解碼影像，速度極快
  • read()：抓取 + 完整解碼成 numpy array，比較耗時

這個技巧在大量跳幀時可節省 30–50% 的影片解碼時間。

# pipeline.py — speed 跳幀
for _ in range(speed - 1):    # 跳過 N-1 幀（只 grab，不解碼）
    cap.grab()
ret, frame = cap.read()       # 第 N 幀才真正讀取

③ Pi 模擬縮放（--pi-sim）

開發機的測試影片解析度可能是 1080p 或 720p，但 Pi Camera 通常輸出 640×480（480p）。

--pi-sim 會在每幀進入偵測之前，先把畫面強制縮到 640×480，讓開發機模擬 Pi 的視野。

  • 實際在 Pi 上跑時不需要這個旗標（Camera 本身就輸出 480p）
  • 只在開發機上「假裝是 Pi 在跑」時用
# pipeline.py
if pi_sim:
    frame = cv2.resize(frame, (640, 480), interpolation=cv2.INTER_LINEAR)

④ 跳幀推論（--skip）

AI 推論是最耗時的步驟（Pi 上約 90ms / 幀）。--skip N 讓程式每 N 幀才呼叫一次 detector，中間幀沿用上次結果。

  • --skip 1（預設）：每幀都推論，最準確
  • --skip 2：推論量減半，FPS 翻倍，框稍微不那麼即時
  • --skip 3：推論量剩 1/3，極限省力模式

視覺上的感覺：輸出影片顯示幀率不變（還是原本的 30fps），但偵測框每 N 幀才「更新」一次——火焰快速移動時框會有輕微延遲感。

# pipeline.py — 跳幀控制
if frame_idx % frame_skip == 0:       # 該推論的幀
    last_detections = detector.detect(frame)
# 否則沿用 last_detections（不呼叫 detector）

⑤ EMA FPS 平滑

即時 FPS 每幀都在抖動（某幀快、某幀慢）。如果直接顯示「瞬時 FPS」，畫面左上角的數字會跳來跳去，很難閱讀。

解法：指數移動平均（EMA）

EMA 公式：新 EMA = α × 瞬時值 + (1 - α) × 舊 EMA

  • α = 0.1（本系統的設定）
  • α 越小 → 顯示越穩定，但反應越慢
  • α 越大 → 反應越快，但數字跳動越明顯
瞬時 FPS
跳動劇烈
8 → 15 → 10
→
EMA 平滑
α = 0.1
→
平滑 FPS
緩慢變化
11.2 → 11.4
# pipeline.py
ema_fps = 0.1 * inst_fps + 0.9 * ema_fps

⑥ 輸出與顯示

每處理完一幀，同時做兩件事：

  • 寫入 mp4：如果有指定 --output，用 cv2.VideoWriter 把這幀附加到輸出影片中。編碼格式：mp4v。
  • 顯示視窗：如果沒有 --no-show，用 cv2.imshow() 把當前幀即時顯示在螢幕上。

使用者可以隨時按 Q 提早結束迴圈（透過 cv2.waitKey(1) 偵測按鍵）。

迴圈結束後（影片播完或按 Q），程式會自動：
・ cap.release()：釋放影片資源
・ writer.release()：關閉輸出檔案（不然 mp4 會損壞）
・ cv2.destroyAllWindows()：關閉所有視窗

最後回傳 avg_fps、total_frames、resolution 等統計數據，由 main.py 印出。
